# 上下文工程与记忆

MyCodeAgent 的上下文工程体系由四个模块协同构成：会话历史记忆、历史压缩归档、会话快照持久化、文件状态追踪与安全写入。

---

## 1. 会话历史记忆（HistoryManager）

`core/context_engine/history_manager.py` 的 `HistoryManager` 是对话记忆的核心数据结构。

**存储形式**：纯内存（`_messages: List[Message]`），不会自动写磁盘。

**消息类型**：

| 角色 | 写入方法 | 触发时机 | 元数据 |
|------|---------|---------|--------|
| `user` | `append_user()` | 用户每次输入 | 开启新轮（round） |
| `assistant` | `append_assistant()` | LLM 每次响应 | `action_type`, `tool_calls`, `reasoning_content`, `step` |
| `tool` | `append_tool()` | 工具执行后 | `tool_name`, `tool_call_id`, 截断后的输出 |
| `summary` | `append_summary()` | 压缩完成后 | `generated_at` 时间戳 |

**关键设计**：

- **轮次识别**（`_identify_rounds`）：以 user 消息为边界切分轮次，summary 不归属任何轮，为压缩提供结构化粒度
- **token 估算**：双路径——精确路径用 API 返回的 `usage.total_tokens` 累加；粗略路径用总字符数 ÷ 3
- **序列化**：`serialize_messages()` 将完整元数据写入可持久化结构，每个消息保存 `{role, content, metadata}` 三字段；`load_messages()` 完整反序列化恢复

## 2. 历史压缩归档

由 HistoryManager + SummaryCompressor + ObservationTruncator 三组件协同。

### 2.1 压缩触发条件（`should_compress`）

- 消息数 ≥ 3
- `estimated_tokens ≥ context_window × compression_threshold`（默认 100k × 0.8 = 80k tokens）
- 轮次数必须超过 `min_retain_rounds`（默认 10 轮），否则拒绝压缩

### 2.2 Summary 生成（`summary_compressor.py`）

- 将待压缩消息序列化为 `[User]/[Assistant]/[Tool]/[Summary]` 格式文本
- 调用 LLM 生成结构化摘要（目标与状态、技术上下文、已完成里程碑、关键洞察、文件系统状态）
- **超时降级**：120 秒超时（`ThreadPoolExecutor + future.result(timeout=120)`），超时返回 `None`，仅做硬截断

### 2.3 压缩执行（`compact`）

1. 提取旧消息（`retain_start_idx` 之前，且 `role != "summary"`）
2. 保留已有 summary + 插入新 summary + 保留最近 N 轮完整消息
3. 旧轮次的原始 user/assistant/tool 消息**永久删除**，不会归档到任何地方
4. 不可拆分原则：tool_use/tool_result 成对保留在同一轮内，边界对齐完整轮次

### 2.4 工具输出截断与归档（`observation_truncator.py`）

**触发时机**：ReAct 循环中，每次工具执行完写入历史时自动触发，无需用户干预。

**阈值**：

- `TOOL_OUTPUT_MAX_LINES = 2000`（行）
- `TOOL_OUTPUT_MAX_BYTES = 51200`（50KB）

**处理流程**：

```
工具返回原始 JSON
  → ObservationTruncator.truncate()
    → 检查：超过 2000 行或 50KB？
      ├─ 否 → 原样返回
      └─ 是 → ① 完整内容落盘到 tool-output/tool_<timestamp>_<toolname>.json
              ② 按 head/tail/head_tail 截断
              ③ 截断后 JSON 返回给 HistoryManager（status 变为 "partial"）
              ④ 返回的 data.truncation.full_output_path 指向归档文件
```

**归档生命周期**：

- 创建：每次工具输出超阈值自动写入
- 清理：10% 概率触发过期检查，删除 7 天前的文件（`TOOL_OUTPUT_RETENTION_DAYS=7`）
- 与快照关系：`/save` 的快照中只记录 `"tool_output_dir": "tool-output"` 字符串引用，不打包归档文件

## 3. 会话快照持久化（`/save` 和 `/load`）

`core/session_store.py` 实现会话的完整快照与恢复。

### 3.1 `/save` — 保存当前一切到单个 JSON 文件

调用链：

```
/save <path>
  → CodeAgent.save_session(path)
    → session_store.build_session_snapshot(...)
    → session_store.save_session_snapshot(path, snapshot)
```

**快照内容**：

```json
{
    "version": 1,
    "system_messages": [...],        // L1+L2 构建结果（含 tools/MCP/Skills 提示词）
    "history_messages": [...],       // 完整对话历史（含 metadata、tool_calls、summary）
    "tool_schema_hash": "sha256...", // 工具 schema 哈希（恢复时校验）
    "read_cache": {...},             // 乐观锁 Read 缓存（路径 → mtime/size）
    "teams_snapshot": {...},         // AgentTeams 协作状态（工作项、消息队列）
    "project_root": "/abs/path",
    "tool_output_dir": "tool-output" // 仅字符串引用，不打包归档文件
}
```

**不包含的内容**：

- 被 `compact()` 丢弃的旧轮次消息
- `tool-output/*.json` 完整归档文件

### 3.2 `/load` — 从 JSON 文件完整恢复

```python
def load_session(self, path):
    snapshot = load_session_snapshot(path)
    self._system_messages_override = snapshot["system_messages"]  # 恢复 system 层
    self.history_manager.load_messages(snapshot["history_messages"])  # 恢复对话历史
    self.tool_registry.import_read_cache(snapshot["read_cache"])  # 恢复乐观锁缓存
    self.team_manager.import_state(snapshot["teams_snapshot"])  # 恢复多 agent 状态
```

恢复后 Agent 等同于 `/save` 那一刻的状态，可继续对话，LLM 看到完整历史上下文和已读文件状态。

### 3.3 多会话支持

**支持多个快照**，path 参数由用户决定：

```
/save                                    → memory/sessions/session-latest.json（默认）
/save memory/sessions/项目A.json          → 自定义路径
/load memory/sessions/项目A.json          → 切换到项目 A 的上下文
```

**无内置会话管理**：没有 `/sessions list`、`/sessions switch` 等命令，多会话靠文件系统管理。

### 3.4 自动保存

三种情况触发自动保存（均写入同一个文件 `session-latest.json`）：

| 触发方式 | 原因标记 |
|---------|---------|
| Ctrl+C / Ctrl+D | `keyboard interrupt` |
| 输入 exit/quit/q | `exit` |
| 正常结束（finally） | `finalize` |

**全部为覆盖写入**，不做合并、追加或检查。同一会话生命周期内 `flag["saved"]` 标记防止重复保存，但不防止下次启动的新会话覆盖旧快照。保留多个会话需手动 `/save` 到不同文件名。

## 4. 文件状态追踪与安全写入

四层防护防止并发写冲突和数据丢失。

### 4.1 第 1 层：Read 时 mtime 追踪（`read_file.py`）

- `ReadTool` 维护类级 `_mtime_cache: {绝对路径 → mtime_ns}`
- 每次读取记录文件当前修改时间
- 检测外部修改：若同文件再次读取时 mtime 变化，标记 `modified_externally: true`

### 4.2 第 2 层：Read → 乐观锁缓存注入（`registry.py`）

Read 成功后，ToolRegistry 自动缓存元信息到 `_read_cache`：

```python
{
    "path_resolved": "...",    // 规范化路径（主键）
    "file_mtime_ms": ...,      // 修改时间（毫秒）
    "file_size_bytes": ...,    // 文件大小（字节）
    "captured_at": ...         // 缓存时间戳
}
```

同时以原始 path 和 resolved path 双键索引。

### 4.3 第 3 层：Write/Edit 前乐观锁自动注入（`registry.py`）

`execute_tool("Write"/"Edit"/"MultiEdit", ...)` 时，若参数缺少 `expected_mtime_ms`/`expected_size_bytes`，框架自动从 `_read_cache` 查找并注入，LLM 无需手动传递。

### 4.4 第 4 层：写入前双重 TOCTOU 校验

**Write 工具**（`write_file.py`）：

- 第一次校验（读取原文件前）：比对当前 `stat()` 的 mtime/size 与期望值
- 第二次校验（写入前，缩小竞态窗口）：再次比对，不匹配返回 `CONFLICT` 错误
- 原子写入：先写临时文件 `file.tmp.<pid>.<timestamp>`，再 `os.replace()`（原子 rename）

**Edit 工具**（`edit_file.py`）：

- 乐观锁校验：未先 Read 直接 Edit → 拒绝（提示必须先 Read）
- 唯一性校验：`old_string` 匹配 0 次报 NOT_FOUND，>1 次报重复次数
- 二进制检测：读前 8KB 检测 `\x00`，拒绝编辑二进制文件
- CRLF/LF 自动适配：统计原文件换行符类型，匹配后还原
- 原子写入：同 Write 的 temp-then-rename 策略

## 整体数据流

```
┌─ 会话运行中 ────────────────────────────────────────────────┐
│                                                              │
│  HistoryManager（纯内存）                                    │
│  ├─ user / assistant / tool / summary 消息                  │
│  └─ compact() → 旧轮次删除，插入 summary                    │
│                                                              │
│  ObservationTruncator → 工具输出超 2000行/50KB 时触发        │
│  └─ tool-output/*.json（完整原始输出，独立于快照，7天过期）  │
│                                                              │
├─ /save 时 ──────────────────────────────────────────────────┤
│                                                              │
│  快照 JSON = system + history(已压缩形态) + read_cache       │
│             + teams + hashes + tool_output_dir(字符串引用)   │
│                                                              │
│  ❌ 不含：被 compact() 丢弃的旧消息                          │
│  ❌ 不含：tool-output/*.json 归档文件内容                    │
│                                                              │
├─ /load 时 ──────────────────────────────────────────────────┤
│                                                              │
│  快照 JSON → 恢复 system / history / read_cache / teams     │
│                                                              │
├─ 会话结束 ──────────────────────────────────────────────────┤
│                                                              │
│  自动保存 → session-latest.json（覆盖写入）                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```
